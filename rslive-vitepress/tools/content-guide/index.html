На странице

Инструкция по созданию контента ​

Эта страница фиксирует рабочий формат для новых материалов RSLive: обычный Markdown, расширения VitePress и подключенные плагины. Для большинства статей достаточно Markdown и frontmatter; Vue-компоненты стоит использовать там, где нужен интерактив, таблицы, графики или специальный виджет.

Шаблон страницы ​

md
---
title: Название страницы
description: Короткое описание для SEO, OpenGraph и поиска.
comment: true
---

# Название страницы

Короткое вступление: что пользователь решит с помощью этой страницы.

## Первый раздел

Текст, ссылки, списки, таблицы, компоненты.

title попадает в заголовок страницы и OpenGraph. description используется в meta description, OpenGraph, Twitter card и поисковом индексе. comment: true включает Giscus, если на деплое заданы переменные окружения Giscus.

Базовый Markdown ​

md
Текст с **жирным**, *курсивом*, `inline code` и [ссылкой](/arrival/start/).

- Пункт списка
- Второй пункт

1. Первый шаг
2. Второй шаг

> Важное замечание в цитате.

| Документ | Когда нужен |
| --- | --- |
| Белый картон | После въезда |
| Боравак | Для долгого пребывания |

Для внутренних ссылок используем абсолютные пути VitePress с завершающим /: /arrival/boravak/. Это ближе всего к DokuWiki-режиму, где : был заменен на /.

Контейнеры VitePress ​

md
::: info
Нейтральное пояснение.
:::

::: tip
Практический совет.
:::

::: warning
Важно проверить перед действием.
:::

::: danger
Риск штрафа, отказа или потери доступа.
:::

::: details Показать подробности
Дополнительная информация, которую не нужно держать открытой всегда.
:::

Контейнеры хорошо подходят для коротких предупреждений, но не заменяют основной текст. Если предупреждение длинное, лучше сделать отдельный раздел.

Сноски ​

Сноски подключены через markdown-it-footnote.[1]

md
Текст со сноской.[^source]

[^source]: Описание источника или пояснение.

Steps ​

Для пошаговых инструкций используем локальный Vue-компонент Steps. Заголовок шага и номер выводятся на одной линии.

md
<Steps>
  <Step title="Подготовить документы">
    Проверьте паспорт, страховку и подтверждение адреса.
  </Step>
  <Step title="Подать заявление">
    Заполните форму и приложите сканы документов.
  </Step>
</Steps>

Mermaid ​

Mermaid подключен через vitepress-plugin-mermaid. Для схем миграции, процессов и зависимостей используем fenced block mermaid.

md
```mermaid
flowchart TD
  Entry[Въезд] --> Karton[Белый картон]
  Karton --> Boravak[ВНЖ]
  Boravak --> Stalni[ПМЖ]
```

Для стабильности лучше использовать латинские id узлов (Entry, Karton), а русский текст писать в квадратных скобках.

Timeline ​

Timeline подключен через vitepress-markdown-timeline. Он полезен для планов на первые дни, истории изменений и дорожных карт.

День 1
  • Оформить регистрацию пребывания.
  • Проверить доступы и документы.
Неделя 1
  • Открыть банковский счет.
  • Подготовить документы для следующего статуса.
md
::: timeline День 1
- Оформить регистрацию пребывания.
- Проверить доступы и документы.
:::

::: timeline Неделя 1

:::

LiteTree ​

LiteTree подходит для дерева документов, маршрутов или структуры раздела. Надежнее передавать дерево через data, чтобы Markdown не съедал отступы.

  • Инструкция/
    • Подготовка/
      • Визы
      • Документы
    • Переезд/
      • Белый картон
      • Банки
    • Интеграция/
      • ПМЖ
md
<LiteTree
  lite
  :data="`Инструкция/
    Подготовка/
        Визы
        Документы
    Переезд/
        Белый картон
        Банки
  `"
/>

Изображения и viewer ​

vitepress-plugin-image-viewer подключен ко всем картинкам внутри .vp-doc. Достаточно обычного Markdown-изображения.

md
![Описание изображения](/images/example.png)

Описание в alt обязательно: оно нужно для доступности, поиска и AI crawlers. Для больших изображений лучше держать файл в docs/public/ и ссылаться от корня сайта.

Комментарии Giscus ​

Giscus подключен в теме, но включается только при наличии переменных окружения:

txt
VITE_GISCUS_REPO=owner/repo
VITE_GISCUS_REPO_ID=...
VITE_GISCUS_CATEGORY=General
VITE_GISCUS_CATEGORY_ID=...

На странице комментарии включаются через frontmatter:

yaml
comment: true

Для справочных страниц, где обсуждения не нужны, ставим comment: false или не указываем поле.

Таблицы, API и графики ​

Простые статичные таблицы можно писать Markdown-таблицей. Большие таблицы с фильтрами, сортировкой или данными из API лучше выносить в Vue-компонент.

md
<DataTable source="/api/exchanges.json" />

Для графиков придерживаемся того же принципа: данные отдельно, визуализация компонентом.

md
<DiasporaChart source="/api/diaspora.json" />

Так контент остается читаемым, а интерактив не превращает статью в набор скриптов.

PWA и PDF ​

PWA работает на уровне сайта: авторам страниц ничего включать не нужно. После сборки появляются manifest.webmanifest, sw.js и registerSW.js.

PDF-экспорт запускается командой:

sh
npm run export-pdf

Настройки экспорта лежат в .vitepress/vitepress-pdf.config.ts. PDF стоит использовать для отдельных релизов гайда, а не как обязательный шаг каждого обновления.

Что не стоит делать ​

  • Не вставлять большие HTML-блоки, если это можно выразить Markdown или компонентом.
  • Не хранить большие данные прямо в статье: таблицы, справочники и графики лучше кормить из JSON/API.
  • Не использовать русские id в Mermaid-узлах, если схема может усложняться.
  • Не делать SEO-текст картинкой: текст должен оставаться текстом.
  • Не писать длинные предупреждения внутри ::: warning; для этого лучше отдельный раздел.
